# Rootstock Remediation Plan

## Objective
Fully remediate the repository issues identified in the audit by executing a correctness-first, low-risk sequence that preserves current functionality, closes semantic bugs, aligns validation and documentation with actual behavior, and leaves the repo with stronger regression coverage and repeatable verification.

## Context
- The repo has three main surfaces:
  - `collector/`: Swift macOS collector producing scan JSON.
  - `graph/`: Python import, inference, reporting, API, viewer, and export pipeline.
  - `docs/`, `examples/`, `scripts/`, `.github/`: public docs, validation, release tooling, CI, and integration harnesses.
- Current verification is green, but several issues remain because existing tests do not assert the right semantics strongly enough.
- The highest-risk problems are in graph identity semantics, inference correctness, and validation/docs drift.
- Historical audit files were moved into `docs/deprecated/audits/`; implementation work should not move them back.

## Execution Principles
1. Fix correctness bugs before cleanup or cosmetic doc work.
2. Keep changes surgical; avoid broad redesign unless required to restore stated architecture.
3. Add or strengthen regression tests before or alongside each behavioral fix.
4. Prefer preserving public contracts when possible; if a contract must change, update all dependent docs/examples/tests in the same iteration.
5. Re-run focused verification after each iteration, then full verification at the end.

## Workstreams

### A. Graph Identity And Topology Correctness
Goal: make scan/host/application identity match the architecture and prevent cross-scan or cross-host corruption.

Scope:
- `graph/import_nodes_core.py`
- `graph/import_nodes_enrichment.py`
- `graph/models.py`
- `graph/diff_scans.py`, `graph/merge_scans.py` if impacted
- graph tests covering import/diff/inference

Tasks:
1. Define the intended identity model explicitly before editing:
   - `Computer` should be keyed by a stable per-scan or per-host composite identity consistent with the architecture.
   - `Application` should not be globally merged by `bundle_id` alone if the same app can exist on multiple hosts or scans.
   - Decide whether application identity is `(scan_id, bundle_id, path)` or `(computer, bundle_id, path)` with scan tagging layered on top.
2. Audit all node `MERGE` patterns in import code and list which currently collapse distinct real-world entities.
3. Update import models and importer code to preserve historical and multi-host correctness.
4. Update enrichment code that currently assumes one global `Application` node set.
5. Correct `LOCAL_TO` inference so it:
   - includes valid users not discovered through unrelated launch-item logic,
   - scopes session-derived relationships to the correct host/scan,
   - avoids cross-host contamination.
6. Add regression tests for:
   - same `bundle_id` on two scans,
   - same `bundle_id` on different hosts,
   - multiple paths for one `bundle_id`,
   - user/group/session host-scoping behavior.
7. Re-run import, diff, and inference suites to confirm no query/report regressions.

Acceptance criteria:
- Re-importing multiple scans no longer overwrites prior application or computer state incorrectly.
- Multi-host scenarios retain distinct topology.
- `LOCAL_TO` edges are correct and host-scoped.
- Diff and merge behavior remains coherent under the new identity model.

### B. Graph Inference Semantics
Goal: make inferred edges, risk categories, and recommendations reflect actual graph topology and model intent.

Scope:
- `graph/infer_file_acl.py`
- `graph/import_nodes_enrichment.py`
- `graph/infer_risk_score.py`
- `graph/infer_recommendations.py`
- any adjacent inference modules whose patterns rely on the same labels/properties

Tasks:
1. Fix file ACL property mismatches between models, importers, and inference:
   - normalize property names,
   - either compute explicit booleans during import or make inference derive them consistently,
   - verify all intended branches can execute.
2. Build a small truth table for each risk-score signal in `infer_risk_score.py`:
   - what source node type emits it,
   - what relationship type should exist,
   - what target node type it points to,
   - what boolean/label predicates are intended.
3. Replace impossible or inverted patterns in risk scoring.
4. Apply the same semantic correction to recommendation generation.
5. Review adjacent inference modules for the same class of bug: relationship direction mismatches, label drift, stale property names, inverted booleans.
6. Strengthen tests so they assert actual scoring/recommendation semantics, not just that properties exist.
7. Confirm idempotency of inference remains intact.

Acceptance criteria:
- Group/world-writable file paths produce expected inferred edges.
- Risk score categories are driven by reachable graph facts, not dead Cypher branches.
- Recommendation edges are produced only when their factual predicates exist.
- Tests fail if relationship direction or property names drift again.

### D. Collector Path And State Semantics
Goal: remove collector false negatives caused by path handling and unknown-state coercion.

Scope:
- `collector/Sources/Entitlements/AppDiscovery.swift`
- `collector/Sources/Quarantine/QuarantineDataSource.swift`
- `collector/Sources/ProcessSnapshot/ProcessSnapshotDataSource.swift`
- `collector/Sources/Firewall/FirewallDataSource.swift`
- `collector/Sources/RemoteAccess/RemoteAccessDataSource.swift`
- related model files and tests

Tasks:
1. Decide and document path semantics for discovered apps:
   - whether to store both original path and resolved path,
   - which path each enrichment stage should use.
2. Fix symlink handling so quarantine reads and process correlation can resolve the actual bundle/executable while preserving original reporting path if needed.
3. Add fixture coverage for symlinked apps spanning discovery, quarantine, and process matching.
4. Refactor firewall posture handling so unreadable/malformed state is represented as unknown/error, not silently as disabled.
5. Refactor remote access collection so command failure is distinguishable from service disabled.
6. Review whether similar “failure => false” coercions exist in adjacent command-backed collector modules and fix the same pattern where security posture could be underreported.
7. Add targeted tests for command-failure semantics.

Acceptance criteria:
- Symlinked apps retain quarantine and process enrichment.
- Collector outputs distinguish collection failure from negative security state.
- Tests cover the new unknown/error semantics.

## Iterative Order
1. Graph identity model and topology fixes.
2. Graph inference semantics and risk/recommendation corrections.

## Verification Strategy

### Per-iteration verification
- Graph-focused iterations:
  - `ruff check .`
  - targeted `pytest` modules for changed areas first
  - broader `pytest` graph suite after each graph iteration
- Collector-focused iterations:
  - targeted `swift test --filter ...`
  - full `swift test` after collector iterations
- Script/docs/tooling iterations:
  - `shellcheck` for changed shell scripts
  - validator/example checks against `demo-scan.json`
  - any doc-link or artifact existence checks added during remediation

### Final verification
- `ruff check .` in `graph/`
- full `pytest` in `graph/`
- full `swift test` in `collector/`
- `shellcheck` on maintained shell scripts
- `docker compose config --quiet` in `graph/`
- run repaired validation command against example scan
- run repaired integration harness or documented smoke equivalent
- surface probe where feasible:
  - graph API endpoint or report generation path using documented config

## Dependencies
- Neo4j test environment available for graph tests that require it.
- Agreement on the intended graph identity model.
- Agreement on canonical collector binary naming.
- Agreement on branch support policy if it affects CI triggers.

## Risks And Mitigations
- Risk: changing graph identity breaks many tests and assumptions.
  - Mitigation: do identity work first, adjust import/enrichment/reporting in one controlled sequence, and add regression fixtures before broad cleanup.
- Risk: validator/schema reconciliation changes public compatibility.
  - Mitigation: prefer aligning code to the repo’s documented and shipped artifacts unless there is a strong security reason to tighten constraints.
- Risk: unknown-state modeling in collector ripples into graph import assumptions.
  - Mitigation: update models/import handling in the same iteration, not later.
- Risk: docs cleanup drifts from implementation again.
  - Mitigation: only update docs after the corresponding behavior is fixed and verified.

## Open Questions
1. **Application identity model:** Should `Application` represent a per-host installation instance, a per-scan observation, or a normalized app entity plus separate observation nodes? Minimal remediation likely favors per-host/per-scan identity without introducing a new observation layer, but this should be decided explicitly before implementation.
2. **Canonical binary name:** Keep public name `rootstock-collector` and adjust scripts/docs/package output references around that, or standardize on `RootstockCLI` internally and externally. Minimal user-facing remediation favors `rootstock-collector`.
3. **Supported branch policy:** Should CI and support target `main`, `dev`, or both. Minimal confusion favors aligning docs and CI on whichever branch is actually used for releases.

## Definition Of Done
- All audited correctness issues are either fixed or explicitly superseded by a documented architectural decision.
- Regression tests exist for every bug class identified in the audit.
- Public validation/docs/examples/scripts are internally consistent and executable.
- Full verification completes successfully.
- Final state can be summarized with `VERDICT: PASS`.
