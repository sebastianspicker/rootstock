Welcome to Rootstock

Graph-based attack path discovery for macOS security boundaries.

Rootstock maps TCC grants, entitlements, Keychain ACLs, code signing metadata, and XPC services as a directed property graph — enabling automated discovery of privilege escalation paths that would be impossible to find manually.

Think: BloodHound for macOS-native security boundaries.

Graph Contents

  • Application — all installed apps with signing metadata
  • TCC_Permission — every TCC service (FDA, Camera, Mic, …)
  • Entitlement — code signing entitlements per app
  • XPC_Service — LaunchDaemons and LaunchAgents
  • Keychain_Item — Keychain ACL metadata (no secrets)
  • MDM_Profile — installed configuration profiles
  • User — local user accounts
  • Computer — scanned host with posture properties
  • CriticalFile — security-critical file ACL metadata
  • BluetoothDevice — paired Bluetooth peripherals
  • CertificateAuthority — signing certificate chain CAs
  • AuthorizationRight — macOS authorization DB rights
  • SystemExtension — network/endpoint security/driver extensions
  • SudoersRule — sudoers file entries
  • LoginSession — active login sessions

Getting Started

Explore the TCC attack surface.

TCC (Transparency, Consent, and Control) is the macOS permission system that controls access to sensitive resources: Full Disk Access, Camera, Microphone, Contacts, and more.

Every TCC grant is a potential attack vector — if an app with Full Disk Access can be injected with attacker code, that code inherits the FDA grant silently.

Try These First

  • Run the query on the right to see all active TCC grants
  • Look for apps with multiple grants — they're the richest targets
  • Check which apps hold kTCCServiceSystemPolicyAllFiles (FDA)

All Active TCC Grants

MATCH (app:Application)-[r:HAS_TCC_GRANT {allowed: true}]->(perm:TCC_Permission)
RETURN app.name         AS app,
       perm.display_name AS permission,
       r.auth_reason     AS granted_by,
       app.is_system     AS is_system
ORDER BY perm.display_name, app.name
LIMIT 50

Apps with the Most TCC Permissions

MATCH (app:Application)-[:HAS_TCC_GRANT {allowed: true}]->(perm:TCC_Permission)
WITH app, collect(perm.display_name) AS permissions
RETURN app.name           AS app,
       app.bundle_id       AS bundle_id,
       app.is_system       AS is_system,
       permissions,
       size(permissions)   AS permission_count
ORDER BY permission_count DESC
LIMIT 20

Find Injectable Apps

Critical: apps with Full Disk Access that can be code-injected.

An app is injectable if it lacks Hardened Runtime or Library Validation — meaning an attacker can load arbitrary code into the process via DYLD_INSERT_LIBRARIES or an unsigned dylib placed on the load path.

If that app also holds Full Disk Access, the injected code inherits FDA — enabling silent read/write of TCC.db, SSH keys, Mail, and all user files.

Injection Methods

  • dyld_insert — missing Hardened Runtime
  • missing_library_validation — accepts unsigned dylibs
  • dyld_insert_via_entitlement — has allow-dyld entitlement
  • electron_env_var — ELECTRON_RUN_AS_NODE abuse

Killer Query 1 — Injectable FDA Apps

MATCH (app:Application)
      -[:HAS_TCC_GRANT {allowed: true}]->
      (:TCC_Permission {service: 'kTCCServiceSystemPolicyAllFiles'})
MATCH (:Application {bundle_id: 'attacker.payload'})-[inj:CAN_INJECT_INTO]->(app)
WITH app, collect(DISTINCT inj.method) AS injection_methods
RETURN app.name            AS app_name,
       app.bundle_id       AS bundle_id,
       app.path            AS path,
       app.team_id         AS team_id,
       injection_methods,
       size(injection_methods) AS method_count
ORDER BY method_count DESC, app.name ASC

Visualise the Attack Surface

MATCH (attacker:Application {bundle_id: 'attacker.payload'})
      -[inj:CAN_INJECT_INTO]->(app:Application)
      -[:HAS_TCC_GRANT {allowed: true}]->(perm:TCC_Permission)
WHERE perm.service = 'kTCCServiceSystemPolicyAllFiles'
RETURN attacker, inj, app, perm
LIMIT 25

Attack Paths

Discover multi-hop privilege escalation chains.

The shortestPath algorithm finds the minimum number of hops from the attacker node to Full Disk Access. A 2-hop path means:

  1. Attacker injects into App A
  2. App A has Full Disk Access

A 3-hop path might involve Apple Event automation or injection chaining.

Tip: Click a path node in the graph to see its full properties. Right-click → "Expand" to explore neighbours.

The attacker.payload node represents any attacker-controlled code on the system. It is created by infer.py.

Killer Query 2 — Shortest Path to Full Disk Access

MATCH (attacker:Application {bundle_id: 'attacker.payload'}),
      (fda:TCC_Permission {service: 'kTCCServiceSystemPolicyAllFiles'})
MATCH p = shortestPath((attacker)-[*..5]->(fda))
RETURN p,
       length(p)                                                AS path_length,
       [n IN nodes(p) | coalesce(n.name, n.display_name, '?')] AS node_names,
       [r IN relationships(p) | type(r)]                        AS rel_types
ORDER BY path_length ASC
LIMIT 10

Injection Chain (Multi-hop)

MATCH path = (attacker:Application {bundle_id: 'attacker.payload'})
             -[:CAN_INJECT_INTO*1..3]->
             (target:Application)-[:HAS_TCC_GRANT {allowed: true}]->(perm:TCC_Permission)
WHERE perm.service IN [
    'kTCCServiceSystemPolicyAllFiles',
    'kTCCServiceAccessibility',
    'kTCCServiceScreenCapture'
]
RETURN path, length(path) AS hops, target.name AS target, perm.display_name AS permission
ORDER BY hops ASC LIMIT 15

Electron Risks

Electron apps inherit TCC permissions in child processes.

When the ELECTRON_RUN_AS_NODE environment variable is set, an Electron app spawns a Node.js REPL that inherits the parent process's TCC grants — including Full Disk Access, Camera, and Microphone.

An attacker with local code execution can exploit this to access protected resources silently, without any user prompt.

Why This Matters

  • Many popular apps use Electron: Slack, VS Code, Discord, 1Password
  • No special privileges required — any user can set env vars
  • TCC grants are inherited by the child Node.js process
  • The OS does not prompt the user when this happens

Killer Query 3 — Electron TCC Inheritance

MATCH (app:Application {is_electron: true})
      -[:HAS_TCC_GRANT {allowed: true}]->(perm:TCC_Permission)
WITH app, collect(DISTINCT perm.display_name) AS inherited_permissions
RETURN app.name              AS app_name,
       app.bundle_id         AS bundle_id,
       inherited_permissions,
       size(inherited_permissions) AS permission_count
ORDER BY permission_count DESC, app.name ASC

Electron Apps + Injection Susceptibility

MATCH (app:Application {is_electron: true})
OPTIONAL MATCH (app)-[:HAS_TCC_GRANT {allowed: true}]->(perm:TCC_Permission)
WITH app, collect(perm.display_name) AS permissions
RETURN app.name                         AS app,
       app.hardened_runtime             AS hardened_runtime,
       app.library_validation           AS library_validation,
       permissions,
       size(app.injection_methods) > 0  AS is_injectable,
       app.injection_methods            AS injection_methods
ORDER BY is_injectable DESC, size(permissions) DESC

Blue Team Audit

Establish a TCC baseline and detect anomalies.

Defenders use Rootstock to audit TCC grants at scale — identifying over-privileged apps, MDM-managed permissions, and unusual entitlements that indicate a supply-chain compromise or malware installation.

Key Questions for Blue Teams

  • Which non-system apps have Full Disk Access?
  • Are there apps with private Apple entitlements?
  • Are any TCC grants MDM-managed (cannot be revoked by user)?
  • Which apps can send Apple Events to privileged apps?
  • Are there unknown LaunchDaemons running as root?

Run the TCC overview query (right) before and after a software install to detect new TCC grants introduced by the installer.

Killer Query 7 — TCC Grant Overview (Blue Team)

MATCH (app:Application)-[r:HAS_TCC_GRANT]->(perm:TCC_Permission)
WITH perm.display_name AS permission,
     perm.service      AS service,
     sum(CASE WHEN r.allowed = true  THEN 1 ELSE 0 END) AS allowed_count,
     sum(CASE WHEN r.allowed = false THEN 1 ELSE 0 END) AS denied_count,
     count(*) AS total_grants
RETURN permission, service, allowed_count, denied_count, total_grants
ORDER BY total_grants DESC

Apple Event TCC Cascade

MATCH (source:Application)-[:CAN_SEND_APPLE_EVENT]->(target:Application)
      -[:HAS_TCC_GRANT {allowed: true}]->(perm:TCC_Permission)
WHERE NOT (source)-[:HAS_TCC_GRANT {allowed: true}]->(perm)
  AND source.bundle_id <> 'attacker.payload'
RETURN source.name        AS sender,
       target.name        AS target,
       perm.display_name  AS permission_gained
ORDER BY source.name, perm.display_name

Private Entitlement Audit

MATCH (app:Application {is_system: false})
      -[:HAS_ENTITLEMENT]->(ent:Entitlement {is_private: true})
WITH app, collect(DISTINCT ent.name) AS private_entitlements
RETURN app.name                          AS app_name,
       private_entitlements,
       size(app.injection_methods) > 0   AS is_injectable,
       size(private_entitlements)        AS private_ent_count
ORDER BY private_ent_count DESC, app.name ASC
LIMIT 20

Tier Classification & Ownership

Prioritise findings by asset criticality.

Rootstock classifies applications into tiers based on their security impact:

  • Tier 0 — Crown jewels: injectable apps with FDA, Accessibility, or Screen Recording. Compromise = full control.
  • Tier 1 — High value: apps with moderate TCC grants or injection paths to Tier 0.
  • Tier 2 — Everything else.

Ownership Workflow

Use mark_owned.py to mark apps you control (e.g., after achieving code execution). Then run blast-radius queries to see what Tier 0/1 assets are reachable from your owned nodes.

Query 46 — Tier Classification Summary

MATCH (app:Application)
WHERE app.tier IS NOT NULL
OPTIONAL MATCH (app)-[:HAS_TCC_GRANT {allowed: true}]->(perm:TCC_Permission)
OPTIONAL MATCH (app)-[:HAS_ENTITLEMENT]->(ent:Entitlement {is_private: true})
WITH app,
     collect(DISTINCT perm.display_name) AS tcc_grants,
     collect(DISTINCT ent.name)          AS private_entitlements,
     size(app.injection_methods)         AS injection_method_count
RETURN app.tier              AS tier,
       app.name              AS app_name,
       app.bundle_id         AS bundle_id,
       app.is_system         AS is_system,
       tcc_grants,
       private_entitlements,
       injection_method_count,
       app.is_electron       AS is_electron
ORDER BY app.tier ASC, app.name

Query 41 — Owned to Full Disk Access

MATCH (src)
WHERE src.owned = true
MATCH (fda:TCC_Permission {service: 'kTCCServiceSystemPolicyAllFiles'})
MATCH p = shortestPath((src)-[:CAN_INJECT_INTO|CHILD_INHERITS_TCC|CAN_SEND_APPLE_EVENT|HAS_TCC_GRANT|CAN_HIJACK|PERSISTS_VIA|CAN_READ_KEYCHAIN|COMMUNICATES_WITH|HAS_TRANSITIVE_FDA|SUDO_NOPASSWD|CAN_WRITE|CAN_MODIFY_TCC|CAN_INJECT_SHELL|ACCESSIBLE_BY|CAN_CONTROL_VIA_A11Y|CAN_BLIND_MONITORING|CAN_DEBUG*..6]->(fda))
RETURN p,
       length(p) AS path_length,
       [n IN nodes(p) | coalesce(n.name, n.display_name, n.bundle_id, '?')] AS node_names,
       [r IN relationships(p) | type(r)] AS rel_types,
       src.owned_at AS owned_since
ORDER BY path_length ASC
LIMIT 10

Physical Security & iCloud

Assess physical and cloud attack surface.

The Computer node stores physical security posture: Lockdown Mode, screen lock, FileVault, Thunderbolt security, Bluetooth discoverability, and Secure Boot level.

iCloud Sync Risks

Injectable apps with iCloud container entitlements can exfiltrate data via iCloud sync. If the host has iCloud signed in, injected code can access synced files, keychain items, and CloudKit containers across all enrolled Apple devices.

Query 64 — Weak Physical Posture

MATCH (c:Computer)
WITH c,
     CASE WHEN c.lockdown_mode_enabled = false OR c.lockdown_mode_enabled IS NULL THEN 1 ELSE 0 END AS no_lockdown,
     CASE WHEN c.bluetooth_discoverable = true THEN 1 ELSE 0 END AS bt_discoverable,
     CASE WHEN c.screen_lock_enabled = false OR c.screen_lock_enabled IS NULL THEN 1 ELSE 0 END AS no_screen_lock,
     CASE WHEN c.filevault_enabled = false OR c.filevault_enabled IS NULL THEN 1 ELSE 0 END AS no_filevault,
     CASE WHEN c.secure_boot_level IS NOT NULL AND c.secure_boot_level <> 'full' THEN 1 ELSE 0 END AS weak_secure_boot,
     CASE WHEN c.external_boot_allowed = true THEN 1 ELSE 0 END AS ext_boot
WITH c,
     no_lockdown + bt_discoverable + no_screen_lock + no_filevault + weak_secure_boot + ext_boot AS weakness_score
WHERE weakness_score > 0
RETURN c.hostname              AS hostname,
       weakness_score,
       c.lockdown_mode_enabled AS lockdown_mode,
       c.bluetooth_discoverable AS bt_discoverable,
       c.screen_lock_enabled   AS screen_lock,
       c.filevault_enabled     AS filevault,
       c.secure_boot_level     AS secure_boot,
       c.external_boot_allowed AS external_boot
ORDER BY weakness_score DESC

Query 68 — Injectable iCloud Sync

MATCH (a:Application)-[:HAS_ENTITLEMENT]->(e:Entitlement {category: 'icloud'})
WHERE size(a.injection_methods) > 0
MATCH (a)-[:INSTALLED_ON]->(c:Computer {icloud_signed_in: true})
RETURN a.name             AS app,
       a.bundle_id        AS bundle_id,
       a.injection_methods AS injection_methods,
       collect(e.name)    AS icloud_entitlements,
       c.hostname         AS hostname,
       c.icloud_drive_enabled    AS drive_enabled,
       c.icloud_keychain_enabled AS keychain_sync
ORDER BY size(a.injection_methods) DESC, a.name

Advanced Attack Paths

File ACLs, shell hooks, and cross-host correlation.

Beyond TCC injection, attackers can escalate via:

  • File ACL abuse — writable TCC.db, sudoers, or LaunchDaemon directories enable direct privilege escalation.
  • Shell hook injection — writable .zshrc, .bashrc, or .zprofile files allow persistent credential theft.
  • Cross-host correlation — when multiple hosts share the same user, compromise of one extends to all.

These queries require the full data collection (file ACLs, shell hooks) to be populated by the collector.

Query 48 — File ACL Write Paths

MATCH (u:User)-[:CAN_WRITE]->(f:CriticalFile)
RETURN u.name       AS user,
       f.path       AS file_path,
       f.category   AS file_type,
       f.owner      AS owner,
       f.mode       AS permissions
ORDER BY f.category, u.name

Query 50 — Shell Hook Injection

MATCH (u:User)-[:CAN_INJECT_SHELL]->(f:CriticalFile {category: 'shell_hook'})
RETURN u.name   AS user,
       f.path   AS hook_path,
       f.owner  AS owner,
       f.mode   AS permissions
ORDER BY u.name, f.path

Certificate & Cloud Risk

Signing integrity and cloud exposure analysis.

Code signing certificates are the trust anchor for macOS security. Weaknesses in the certificate chain reduce the barrier for code injection and app replacement attacks.

Certificate Risk Factors

  • Expired certificates — apps may bypass Gatekeeper re-validation checks.
  • Ad-hoc signatures — no identity verification, trivially forgeable.
  • Non-Apple CA chains — enterprise or third-party CAs may have weaker issuance controls.

Query 60 — Expired Cert with TCC

MATCH (a:Application)-[:HAS_TCC_GRANT {allowed: true}]->(t:TCC_Permission)
WHERE a.is_certificate_expired = true
  AND a.is_sip_protected = false
WITH a, collect(DISTINCT t.service) AS tcc_grants
RETURN a.name                   AS app_name,
       a.bundle_id              AS bundle_id,
       a.certificate_expires    AS certificate_expires,
       a.signing_certificate_cn AS signing_certificate,
       tcc_grants
ORDER BY size(tcc_grants) DESC

Query 62 — Non-Apple CA Chain

MATCH (a:Application)-[:SIGNED_BY_CA]->(leaf:CertificateAuthority)
OPTIONAL MATCH path = (leaf)-[:ISSUED_BY*0..]->(root:CertificateAuthority {is_root: true})
WITH a, leaf, root
WHERE root IS NOT NULL
  AND root.common_name IS NOT NULL
  AND NOT root.common_name STARTS WITH 'Apple'
RETURN a.name                   AS app_name,
       a.bundle_id              AS bundle_id,
       a.signing_certificate_cn AS signing_certificate,
       root.common_name         AS root_ca,
       root.organization        AS root_org
ORDER BY a.name
